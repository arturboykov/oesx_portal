# 1\. Стратегическое позиционирование

## Главный продуктовый тезис

**OES X - среда автоматизации процессов самими сотрудниками**

Каждый сотрудник компании получает персональное AI-пространство, в котором может собирать автоматизации своих рабочих процессов без участия ИТ-разработчиков.

Решения, созданные одним сотрудником, могут быть опубликованы в общий маркетплейс компании и переиспользованы другими.

OES X - это интерфейс настройки и управления всеми решениями: одна точка входа, единая модель безопасности, единый биллинг.

## Стратегическая цель: AI-Native снизу-вверх

«Снизу-вверх»: каждый сотрудник самостоятельно автоматизирует свои регулярные задачи через инструмент, который ему доступен здесь и сейчас. Решения накапливаются, лучшие распространяются через маркетплейс. Компания становится AI-Native не через решение начальства, а через накопление сотен мелких автоматизаций.

**OES X - инструмент именно этой модели.** К концу внедрения OES X в компании-пилоте (ВГК): сотни активных пользователей с собственными агентами, дашбордами и автоматизациями, маркетплейс с десятками переиспользуемых решений, измеримая экономия времени на регулярных задачах.

# 2\. Целевые роли и ценность

OES X 1.0 одновременно обслуживает четыре роли в компании, у каждой - свой контракт на ценность.

## Роль 1: Citizen developer (рядовой сотрудник)

**Кто:** сотрудник любого подразделения - финансы, логистика, HR, операционка, продажи. Не имеет навыков SQL, программирования, RPA-инструментов.

**Боль:** регулярные ручные задачи (сбор отчётов, мониторинг, рассылки, проверки), на которые уходят часы в неделю. Заявки в ИТ или BI идут неделями.

**Ценность OES X:** за минуты собрать рабочую автоматизацию или дашборд в своём персональном пространстве. Использовать готовые решения коллег из маркетплейса.

**Ключевая метрика для роли:** среднее число активных решений на пользователя; время на типовую задачу до и после внедрения.

## Роль 2: Power user / Builder подразделения

**Кто:** продвинутый сотрудник в подразделении, который собирает автоматизации не только для себя, но и для команды.

**Боль:** внутри команды одни и те же задачи делаются по-разному. Хочется стандартизировать, но нет инструмента.

**Ценность OES X:** опубликовать своё решение в общий маркетплейс. Получить признание. Видеть аналитику использования.

**Ключевая метрика:** число опубликованных решений; число форков из маркетплейса.

## Роль 3: ИТ-администратор корпорации

**Кто:** ответственный за безопасность, доступы, биллинг, подключение источников данных.

**Боль:** shadow AI - сотрудники сами заводят учётки в ChatGPT, данные утекают. Альтернативно - приходится самим строить всю автоматизацию.

**Ценность OES X:** одна управляемая платформа вместо shadow AI. RBAC, аудит, лимиты, изоляция данных.

**Ключевая метрика:** снижение запросов в ИТ; полнота audit log; отсутствие security-инцидентов.

## Роль 4: Бизнес-руководитель / спонсор внедрения

**Кто:** CIO/CDO/CEO компании-заказчика.

**Боль:** AI-стратегия требуется, но непонятно как её реализовать без многомесячных проектов.

**Ценность OES X:** путь к AI-Native компании за месяцы, а не годы. Bottom-up адопшн вместо тяжёлого top-down проекта.

**Ключевая метрика:** % сотрудников с активным использованием; число автоматизаций; экономический эффект.

# 3\. Концептуальная архитектура

## Главная архитектурная гипотеза

**Personal Workspace as a Service**

Каждый зарегистрированный сотрудник получает свой постоянно работающий (always-on) экземпляр OpenClaw - Personal Workspace.

Это его персональное AI-пространство: с памятью, с подключенными к нему данными, с собранными им решениями, с историей взаимодействий.

OES X выступает control plane над всеми Personal Workspaces: создание, настройка, изоляция, governance, биллинг, маркетплейс.

Решения создаются в Personal Workspace, опционально публикуются в общий Marketplace, переиспользуются другими сотрудниками - каждым в своём Workspace.

## Аналоги в индустрии

Архитектура per-user workspace становится стандартом для enterprise-AI-платформ в 2026:

- **OpenAI Workspace Agents (апрель 2026):** сдвиг от session-based к persistent workspaces с памятью, файлами, инструментами; admin controls; team directory как общий каталог.
- **Google Gemini Enterprise Agent Platform (апрель 2026):** secure-by-design Workspaces - sandbox-окружение per agent; Agent Identity (cryptographic ID); Agent Gateway для безопасных межагентных взаимодействий.
- **Salesforce Agentforce:** персональные spaces + общий AgentExchange с governance-валидированными решениями.

**Это не «эксперимент с архитектурой»,** а индустриальный mainstream 2026 года. Мы делаем то же самое, но в российском контексте, с собственным стеком и фокусом на citizen developer-а.

## Высокоуровневая структура

Система делится на три плоскости:

- **Control Plane (OES X core):** управляет жизненным циклом Personal Workspaces, governance, маркетплейсом, биллингом, аналитикой.
- **Workspace Plane (Personal Workspaces):** пул индивидуальных AI-пространств сотрудников. Каждое всегда работает, изолировано.
- **Data Plane (Data Layer и интеграции):** общая инфраструктура подключения к корпоративным данным и системам.

**Декомпозиция продукта на 11 функциональных блоков** (см. раздел 5) - это разрез работы команды по функциональности, ортогональный плоскостям архитектуры. Один блок может затрагивать несколько плоскостей, один компонент архитектуры может попадать в несколько блоков.

# 5\. Функциональная декомпозиция продукта на 11 блоков

_Это разрез работы команды по функциональности - что строим и что развиваем как продукт._

Каждый блок имеет владельца и план развития. Блоки нумеруются с 0 - это операционная база, которая должна быть до всего остального.

**Блок 0. CI/CD, версионность, uptime**

**Назначение:** Операционная основа продукта: автоматическая сборка, тестирование, выкатка. Версионирование релизов с возможностью отката. Гарантии доступности.

**В версии 1.0:**

- CI/CD пайплайны на все компоненты системы. Автотесты обязательны для всех новых модулей
- Версионирование артефактов: контейнеров, скилов, шаблонов решений в маркетплейсе. Возможность отката
- SLA по uptime: 90% для Personal Workspaces, 95% для control plane (OES X)
- Мониторинг и алертинг на критичные метрики. Дежурные DevOps

**Связь с компонентами архитектуры:** _Все компоненты архитектуры (поперечная задача)_

**Владелец блока:** Антон Роман

**Блок 0.1. Техподдержка**

**Назначение:** Поддержка пользователей и ИТ-администраторов корпораций. Регламент обращений, эскалаций, SLA на ответы.

**В версии 1.0:**

- Канал поддержки для пользователей (внутри ЛК + email)
- База знаний и FAQ - публикуется параллельно с релизами функциональности
- Регламент эскалаций: какие обращения идут в ТП, какие в продакт, какие в разработку
- Метрики ТП: % обращений без эскалации к разработчикам (целевая ≥ 80%)

**Связь с компонентами архитектуры:** _User Portal, Admin Console_

**Владелец блока:** Ирина Карху

**Блок 1. Админка**

**Назначение:** Центральный интерфейс ИТ-администратора корпорации: подключение пользователей и данных, управление затратами, аудит, централизованные обновления Workspaces.

**В версии 1.0:**

- Подключение пользователей: через корпоративный SSO (Keycloak), массовое заведение, привязка к ролям
- Подключение данных: интерфейс добавления источников (БД, API, MCP, документы). Семантическое описание. Тестирование подключения
- Управление затратами: биллинг по подразделениям и ролям. Лимиты. Отчёты. Прогноз потребления
- Полный лог действий агентов: фильтры по пользователю, времени, типу действия.
- Workspace Manager: централизованные обновления скилов, шаблонов, кастомизаций на все Workspaces одной командой

**Связь с компонентами архитектуры:** _Admin Console, Identity & Access, Billing & Limits, Governance & Audit, Workspace Manager_

**Владелец блока:** Алексей Котов

**Блок 2. Уровень данных**

**Назначение:** Самостоятельное подключение источников данных со стороны ИТ-департамента корпорации. Быстрое подключение, поддержка MCP, требования к семантическим описаниям, open metadata.

**В версии 1.0:**

- Быстрое подключение источников: через UI в Admin Console, не через тикеты в команду OES
- Поддержка MCP-серверов как первоклассного протокола интеграции
- Стандартизированный формат семантического описания: что в таблице, на бизнес-языке, связи между таблицами
- Open metadata: возможность экспорта/импорта семантических карт. Совместимость с открытыми стандартами
- Двухуровневая модель доступа: роли как база + оверрайды на уровне источника
- Связка с Active Directory / Keycloak для автоматического пересчёта прав при изменении должности

**Связь с компонентами архитектуры:** _Data Layer, Identity & Access, Admin Console_

**Владелец блока:** Антон Роман

**Блок 3. Анонимизатор данных**

**Назначение:** Защита чувствительных корпоративных данных от утечек во внешние LLM через маскирование.

**В версии 1.0:**

- Подмена чувствительных значений на моки перед отправкой в LLM, с сохранением типа и структуры данных
- Конфигурация правил маскирования через Admin Console: ИТ задаёт какие поля считать чувствительными
- Восстановление реальных значений в финальном результате - пользователь видит корректные данные
- Работа на любом LLM-провайдере, включая внешние

**Связь с компонентами архитектуры:** _Security Data Agent, Data Layer, Model Router_

**Владелец блока:** Серафим Ситников

**Блок 4. Движок (OpenClaw): данные → value**

**Назначение:** Ядро превращения данных в ценность для пользователя - генерация решений (дашбордов, агентов, автоматизаций) через шаблоны OpenClaw с графом подшаблонов под конкретные функции.

**В версии 1.0:**

- Шаблоны OpenClaw - преднастроенные конфигурации под типовые задачи (создание дашборда, Q&A, алерт, автоматизация)
- Граф подшаблонов под конкретные функции: каждый шаблон - это композиция более мелких подшаблонов
- Управление шаблонами через Admin Console: добавление, обновление, версионирование
- Распространение обновлений шаблонов на все Personal Workspaces через Workspace Manager
- Связь с Marketplace: опубликованное решение пользователя может стать корпоративным шаблоном

**Связь с компонентами архитектуры:** _Personal Workspace, Solution Builder, Workspace Generator, Marketplace_

**Владелец блока:** Алексей Котов

**Блок 5. Организация памяти юзера**

**Назначение:** Персональная память Workspace, которая делает его «своим» для каждого сотрудника. Помнит контекст, предпочтения, историю.

**В версии 1.0:**

- Long-term память: накопленный контекст работы пользователя
- Контекст сессии: история текущего диалога, активные задачи
- Выученные предпочтения: форматы, стилистика, частые источники данных
- Управление памятью пользователем: возможность просмотреть, отредактировать, удалить элементы
- Гарантии приватности: память одного пользователя недоступна другому

**Связь с компонентами архитектуры:** _Memory Subsystem, Personal Workspace_

**Владелец блока:** Илья Гречко

**Блок 6. Подключение к разным интерфейсам (Teams, почта, Max, TG)**

**Назначение:** Дополнительные каналы взаимодействия пользователя с его Personal Workspace помимо веб-ЛК.

**В версии 1.0:**

- Teams-бот: запросы и ответы в чате, привязан к учётке через O-Auth
- Email-канал: отправка запросов на специальный адрес, ответ приходит письмом
- Max и Telegram: дополнительные мессенджеры для российского контекста
- Единый Workspace для всех каналов: память и история общие, не разделены по каналам
- Авторизация: один сотрудник = одна сущность независимо от канала

**Связь с компонентами архитектуры:** _Channels, Identity & Access, Personal Workspace_

**Связь с компонентной архитектурой (раздел 4)**

Функциональные блоки - это работа команды по продукту. Компоненты - это runtime-сущности в системе.

Один функциональный блок может затрагивать несколько компонентов: например, блок «Уровень данных» работает с Data Layer, Identity & Access и Admin Console одновременно.

Один компонент может попадать в несколько блоков: например, Personal Workspace задействован в блоках 4, 5, 6, 7.

При планировании работ команды используется функциональная декомпозиция (11 блоков). При проектировании технических контрактов - компонентная архитектура. Эти проекции дополняют друг друга, не заменяют.

# 6\. Функциональность версии 1.0

## Возможности для citizen developer

- **Личный кабинет** с обзором: мои решения, мой Workspace, мои данные, моё потребление, маркетплейс.
- **Создание решений через диалог:** сотрудник пишет на естественном языке. Система проводит по сценарию (уточнения, выбор источников, превью), собирает артефакт.
- **Полу-ручной режим:** перед сохранением каждое решение проходит через превью с объяснением. Кнопки: «Принять», «Доработать с комментарием», «Отказаться».
- **Типы решений в 1.0:** дашборды (приоритет №1, флагман качества), простые агенты-ассистенты, уведомления и алерты, простые автоматизации.
- **Доработка существующих решений:** выбрать решение → описать что поменять → превью → принять.
- **Каналы взаимодействия:** ЛК (web), Teams-бот, email. Все каналы - единый Workspace. Max и TG - кандидаты на 1.0 или 1.1 (см. блок 6).
- **Память и контекст:** Workspace помнит историю работы, подключенные источники, предпочтения по форматам.

## Возможности маркетплейса

- **Публикация решения:** сотрудник делает решение публичным с описанием, тегами, категорией. Опционально проходит модерацию ИТ.
- **Поиск и просмотр:** сотрудники видят каталог с поиском, фильтрами, рейтингом.
- **Форк (создание копии):** понравившееся решение копируется в Personal Workspace получателя, где он адаптирует под себя.
- **Аналитика для автора:** сколько раз скопировали, кто пользуется, оценки.
- **Шаринг по ссылке:** альтернатива публикации - отправить ссылку конкретному коллеге.

## Возможности ИТ-администратора

- **Управление пользователями и ролями:** создание учёток через SSO, назначение ролей, установка лимитов.
- **Управление источниками данных:** подключение БД, API, MCP, документов. Семантические описания. Open metadata. Тестирование.
- **Окружения (Environments):** создание окружений под подразделения. Каждое со своими источниками и политиками.
- **Модерация маркетплейса:** настраиваемые политики публикации.
- **Аудит и observability:** полный лог действий, алерты на аномалии, экспорт в SIEM.
- **Аналитика использования:** активные пользователи, потребление по подразделениям, топ-решения.
- **Workspace Manager:** централизованные обновления скилов и шаблонов на все Workspaces.
- **Управление API-ключами LLM-провайдеров:** корпорация может использовать свои ключи (pass-through биллинг).

## Возможности безопасности

- **Изоляция Workspaces:** каждый Workspace полностью изолирован.
- **Permission intersection:** решение действует в пересечении прав агента и прав пользователя.
- **Маскирование чувствительных данных:** Security Data Agent заменяет чувствительные значения на моки перед отправкой в LLM.
- **Защита от prompt injection:** проверки входящих данных и инструкций.
- **Полный audit trail:** каждое действие любого агента логируется.

# 7\. Сценарии использования

## Сценарий 1: Создание дашборда (флагман качества)

**Аналитик финансового департамента создаёт мониторинг отгрузок**

1\. Сотрудник заходит в ЛК через корпоративный SSO. Видит свой Personal Workspace с историей.

2\. Нажимает «Создать решение» → выбирает тип «Дашборд».

3\. Пишет: «Покажи мониторинг отгрузок угля за 6 месяцев - динамику, топ направлений, сравнение с прошлым годом».

4\. Workspace через диалог уточняет: имя дашборда, типы графиков, частоту обновления.

5\. Через 30-180 секунд видит превью + объяснение + использованные таблицы.

6\. Принимает → дашборд сохраняется в Workspace, появляется в «Мои решения».

7\. Решает поделиться → копирует ссылку, отправляет коллеге.

8\. Через неделю замечает популярность → публикует в Marketplace. Через месяц 12 коллег сделали форк.

## Сценарий 2: Использование чужого решения через Marketplace

**HR-специалист переиспользует дашборд по найму**

1\. Заходит в Marketplace, ищет «найм» → находит решение коллеги.

2\. Видит описание, рейтинг, скриншот.

3\. Нажимает «Создать копию для себя» → артефакт копируется в его Personal Workspace.

4\. Workspace проверяет права на источники. Если нет - сообщает каких прав не хватает и кому обратиться.

5\. После получения прав адаптирует копию под своё подразделение.

6\. Решение работает как часть его Personal Workspace.

## Сценарий 3: Простая автоматизация через Teams

**Логист настраивает уведомления о просрочках**

1\. В Teams пишет боту: «Каждое утро в 9:00 присылай мне просроченные доставки за вчера, если их больше 5».

2\. Бот уточняет источники данных и критерии.

3\. Создаёт автоматизацию в Personal Workspace, отправляет превью.

4\. Сотрудник принимает → автоматизация работает каждое утро.

5\. Через месяц система предлагает: «Это часто запускают коллеги похожих ролей. Опубликовать в Marketplace?».

## Сценарий 4: ИТ-администратор расследует инцидент

**Security-инженер проверяет утечку данных**

1\. Получает алерт: «Высокая активность к чувствительной таблице из Workspace user-123».

2\. Открывает Admin Console → Audit → фильтр по user-123.

3\. Видит полный лог: какие решения запускались, какие данные, какие промпты ушли в LLM, что вернулось.

4\. Обнаруживает legitimate use case (новый дашборд для финансового отчёта). Закрывает инцидент.

5\. При нарушении - мог бы заблокировать Workspace и отозвать конкретные решения одной командой.

# 8\. Безопасность и Governance

## Принципы

- **Изоляция по умолчанию.** Personal Workspace другого пользователя для меня недоступен ни напрямую, ни через данные, ни через память.
- **Permission intersection.** Effective Action Scope = Agent Permissions ∩ User Permissions. Решение из Marketplace не получает прав автора - работает только с моими.
- **Маскирование чувствительных данных.** Перед отправкой в внешний LLM реальные значения заменяются моками. Маскирование на уровне Security Data Agent (блок 3).
- **Полный audit trail.** Каждое действие любого агента логируется. Структурировано для интеграции с корпоративным SIEM.
- **Защита от prompt injection.** Проверки входящих данных (документов, писем, веб-страниц).
- **Read-only по умолчанию.** Все подключения к корпоративным данным - read-only. Запись только через явные коннекторы с дополнительной верификацией.

# 9\. Биллинг и ценовая модель

## Логика биллинга в per-user модели

Главный риск per-user always-on модели - неконтролируемый рост затрат. Без жёстких лимитов затраты могут вырасти непредсказуемо.

**Решение:** лимиты по ролям и подразделениям. ИТ задаёт лимит токенов в месяц на роль. Сотрудник работает в рамках лимита. Превышение - мягкая остановка с уведомлением.

## Структура лимитов

| **Уровень**                 | **Что лимитируется**              | **Кто настраивает**                                |
| --------------------------- | --------------------------------- | -------------------------------------------------- |
| Корпорация (тенант)         | Общий месячный лимит токенов      | Уровень контракта с OES                            |
| Подразделение / окружение   | Лимит на подразделение            | ИТ-администратор в Admin Console                   |
| Роль                        | Лимит на пользователя данной роли | ИТ-администратор. Пресеты от OES (low/medium/high) |
| Индивидуальный пользователь | Переопределение лимита роли       | ИТ по запросу руководителя сотрудника              |

## Прозрачность

- **Пользователь видит** в ЛК свой текущий лимит, потребление, прогноз исчерпания.
- **Уведомления:** при 80% лимита - предупреждение, при 100% - мягкая остановка.
- **Не видит денежных сумм,** только токены - снимает страх «неожиданного счёта».
- **ИТ видит в Admin Console:** потребление в реальном времени, аномалии, прогноз на конец месяца.
- **Pass-through биллинг:** корпорация может использовать свои API-ключи LLM-провайдеров.

# 11\. Открытые вопросы и риски

## Технические открытые вопросы

- **Изоляция Workspaces:** контейнеризация / namespace / процесс? Требует архитектурной сессии.
- **Хранение состояния Workspace:** формат, схема, ретенция, бэкапы. Что при увольнении сотрудника.
- **Обновления Workspace без даунтайма:** rolling update, blue-green? Через Workspace Manager (блок 1).
- **Холодный старт Workspace для нового сотрудника:** целевая <30 сек до первого ответа.
- **Permission inheritance в маркетплейсе:** как пользователь получает доступ к источникам форкнутого решения.

## Продуктовые открытые вопросы

- **Модерация маркетплейса:** обязательная / опциональная / per-категория. В 1.0 предлагается опциональная.
- **Solution lifecycle:** что с решением при увольнении автора.
- **Версионирование в маркетплейсе:** автор обновил - что у тех, кто форкнул.
- **Объективные метрики success rate:** как мерить когда «правильность» субъективна.
